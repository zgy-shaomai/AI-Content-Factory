# 审核工作台设计（Audit Workflow）

> 项目：AI 内容工厂
> 入口：飞书多维表 + N8N
> 适用 SKU：YN-BRA-001（黑色高弹速干运动内衣 / 前拉链款）
> 文档版本：v1.0 · 2026-05-05

---

## 1. 设计目标

审核工作台承担"AI 生成结果 → 人工把关 → 入库 / 二次生成 / 退回"的所有人机交互。设计原则：

1. **零学习成本**：审核员不接触 N8N、不接触 Postgres、不写命令；全在飞书多维表上完成。
2. **每条候选只做一次决策**：通过 / 打回 / 二次生成，三选一，不允许"待定"长期挂着。
3. **二次生成的反馈结构化**：把驳回意见拆成 `驳回类别 + 改向 + 保留元素`，让 LLM 能直接拿去重写 prompt。
4. **批量与单条都顺手**：单条精修 + 批量同意 + 批量重发，三种节奏共存。
5. **SLA 可见**：超时不审的任务必须主动推送，避免任务堆积。

## 2. 飞书多维表布局

审核工作台基于 **一张多维表（`审核台-内容工厂`） + 三个视图** 完成。

### 2.1 表字段总清单

| 字段名 | 类型 | 来源 | 是否审核员可改 |
|---|---|---|---|
| `任务ID` | 文本（自动） | N8N 写入 | 否 |
| `SKU` | 单选（产品库） | N8N 写入 | 否 |
| `产品名` | 文本 | N8N 写入 | 否 |
| `链路类型` | 单选（图片 / 视频 / 分镜图） | N8N 写入 | 否 |
| `候选编号` | 数字（1~N） | N8N 写入 | 否 |

## 3. 流程：通过 / 打回 / 二次生成

### 3.1 三个动作 → 三个 N8N webhook

| 审核员动作 | 飞书多维表触发 | N8N 入口（webhook） | N8N 后续 |
|---|---|---|---|
| 设状态 = `通过` | 字段变更触发自动化 | `POST /webhook/audit/approve` | 把候选标记为 `delivered`，写 `archive` 表，复制原图到交付路径 `/{tenant}/{sku}/_delivered/`，关闭同任务其他候选 |
| 设状态 = `打回` | 字段变更触发自动化 | `POST /webhook/audit/reject` | 该候选状态变 `rejected`；如该任务 4 个候选全部打回，自动触发"整轮失败"告警，提示是否要重新跑 |
| 设状态 = `二次生成` | 字段变更触发自动化 | `POST /webhook/audit/regenerate` | 读取 `驳回类别 + 改向描述 + 保留元素` 拼成结构化反馈 → 调 LLM 改写 prompt → 提交新一轮生成（轮次 +1） |

每个 webhook 的 payload 都包含：
```json
{
  "task_id": "T-2014",
  "candidate_id": "C-2",
  "sku": "YN-BRA-001",
  "actor": "auditor@client.com",
  "decision": "regenerate",
  "reject_categories": ["模特", "拉链/细节"],
  "redirect": "模特换成偏运动健身风格，拉链头改成银色金属。",
  "preserve": "黑色主色、工作室白底、正面3/4角度。"
}
```

### 3.2 驳回类别（固化为多选枚举）

为了让 LLM 改写 prompt 时能稳定理解，驳回类别**必须从下拉选择**，不允许审核员自己写：

- `模特`：人物形象、姿态、表情、肤色、年龄段
- `面料质感`：纹理、光泽、厚度、垂坠感
- `光线`：色温、方向、强弱、阴影
- `背景 / 场景`：底色、道具、环境
- `拉链 / 细节`：扣件、缝线、印花、Logo
- `比例 / 构图`：取景、角度、人物在画面占比
- `文字 / 水印`：误生成的文字或水印
- `色彩偏移`：成品颜色与产品实物不一致
- `其他`：以上都不属于

### 3.3 二次生成的提示词改写规则

N8N 收到 `regenerate` 后，调 LLM 用如下模板改写 prompt：

```
原 prompt：{original_prompt}
驳回类别：{reject_categories}
改向描述：{redirect}
保留元素：{preserve}

请输出新版 prompt，要求：
1. 保留"保留元素"中提到的所有特征。
2. 针对"驳回类别"和"改向描述"做定向修改，其他部分尽量不变。
3. 不要新增审核员未提及的元素。
```

新 prompt 写回 `tasks.prompt_v{n+1}`，进入新一轮生成；旧候选标记 `superseded`。

## 4. 批量操作策略

### 4.1 批量同意（最常用）

- 在"待审队列"视图中，多选若干行 → 批量改字段 `审核状态 = 通过`。
- 飞书多维表的"批量字段更新"会逐行触发自动化，N8N 顺序处理。
- 实测 50 行批量通过约 30 秒内全部归档完毕。

### 4.2 批量打回（同任务整组否决）

- 场景：一个任务的 4 个候选都不行，准备整轮重做。
- 操作：在视图里筛选 `任务ID = T-2014`，全选 → 批量改 `审核状态 = 二次生成` + 在第一行写改向描述 → 用"复制单元格到所选"把改向同步到其他行。
- N8N 收到 4 条 regenerate 后会做合并：同一 task_id 在 30 秒内的多次 regenerate 合并成一次 prompt 改写 + 一次新一轮生成。

### 4.3 单条精修

- 适合"4 张图里 3 张通过、1 张要小调"的情形。
- 单条改 `审核状态` 即可，不影响其他候选。

### 4.4 批量操作的护栏

- 单次批量上限 100 行（飞书自动化频率限制）。
- 跨链路类型不允许批量（图片 + 视频 不同处理逻辑）。
- 已交付状态不允许回滚（如需作废，走"标记下架"备注 + 工单）。

## 5. 审核 SLA 与告警

### 5.1 SLA 定义

| 指标 | 阈值 | 计时起点 | 计时终点 |
|---|---|---|---|
| 单候选审核响应 | ≤ 4 工作小时 | N8N 写入待审 | 状态变更为非"待审" |
| 二次生成轮数上限 | ≤ 3 轮 | 第 1 轮生成 | 状态变"通过"或"打回" |
| 整任务交付时长 | ≤ 2 工作日 | 任务创建 | 至少 1 个候选状态变"已交付" |

### 5.2 告警触发与推送

N8N 用 cron 定时器（每 30 分钟扫一次）检查超时任务：

```
SELECT task_id, sku, candidate_id, created_at
FROM tasks
WHERE status = 'pending_review'
  AND created_at < NOW() - INTERVAL '4 hours';
```

命中后通过飞书机器人 webhook 推送到 `审核台告警群`：

```
【审核超时】
任务 T-2014  ·  YN-BRA-001
候选 #2 已等待 4.5 小时未审核。
快速跳转：https://feishu.cn/base/xxx?row=...
```

### 5.3 告警分级

- **黄色（4-8 小时未审）**：每 1 小时推送一次，群里 @ 当班审核员。
- **红色（>8 小时未审）**：单独推送给审核负责人 + @ 全员。
- **熔断（>24 小时未审）**：N8N 自动把该候选状态改为 `auto_skipped`，写入告警日志，等人工兜底。

### 5.4 周报指标

每周一上午 9 点 N8N 推送上一周审核数据：通过率、平均响应时长、平均轮数、二次生成命中率、各驳回类别 TOP3。用于复盘 prompt 模板与归类规则。

## 6. 与其他模块的对接

- **生成模块**：N8N 写入候选时一次性创建多维表行，所有候选共享 `task_id`，候选编号从 1 开始。
- **归档模块**：审核状态变"通过"时触发 OSS 复制（详见 `archive-structure.md`）。
- **交付模块**：每个 SKU 累积到既定数量后由审核负责人在飞书点"打交付包"按钮，触发 `webhook/delivery/build`，由 N8N 打 zip 上传 OSS 并把链接回写到飞书。
