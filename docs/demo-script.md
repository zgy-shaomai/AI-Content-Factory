## 2. 客户可能问的 12 个问题 + 标准答复

### Q1：AI 会幻觉，生成的产品和真实款式不一致怎么办？

**答**："我们做了三层防御：第一层，每张候选图生成时强制以你提供的白底产品图作为 reference，模型看着实物画；第二层，prompt 里硬编码产品关键特征（前拉链 / 黑色 / 高弹），模型不能擅自换；第三层，审核员把关，'拉链头不对'这种细节驳回会进入二次生成，结构化地修正。三层叠加之后，幻觉率经我们样品调试可以压到 10% 以下，剩下的由审核员兜底。我们承诺的是'让模型尽量贴近实物 + 让审核能高效拦截偏差'，不是'消除幻觉'——这是行业目前的物理上限。"

### Q2：版权问题？AI 生成的图我能商用吗？

**答**："本项目用的是 5dock NewAPI 转发的商业可用模型，火山方舟 Seedance 也是商用授权。生成的内容版权归调用方（也就是你们公司）。我们会在交付时附一份《模型使用条款摘要》，说明每个模型的可商用范围。如果未来你们要拿生成图去申请外观专利或商标，那不在 AI 生成内容的常规商用范围内，需要单独走法务。"

### Q3：模特看起来不像真人，比例不对，怎么解决？

**答**："这是 AI 生成图最常见的问题，叫 anatomy distortion。我们有几个具体手段：a) prompt 里写明'亚洲女性、25-30 岁、运动健身体型'，比模糊词'beautiful model'稳定得多；b) 强制 reference 你提供的实拍模特图（如有）；c) 用最新代的 SDXL / Flux 模型，比早期 SD 稳定 3-5 倍；d) 4 选 1 的候选机制本身就是把不达标的过滤掉。如果对模特一致性要求极高（比如要做'同一个虚拟代言人'），需要单独训 LoRA，那是另外的项目。"

### Q4：面料质感能还原吗？特别是这种高弹速干面料的光泽。

**答**："分两层。表面层光泽（哑光、亮面、丝光）模型可以稳定还原，靠 prompt 里加 'matte fabric' / 'shiny lycra' 这类词。**纹理层**（华夫格、罗纹的几何纹理）模型还原度大概 70-80%，剩下 20% 需要审核员把关。如果对面料纹理 100% 还原，目前 AI 做不到，需要走传统拍摄+后期。我们的样品 YN-BRA-001 是相对简单的速干面料，验收时你们可以直接对比效果。"

### Q5：产能多少？能保证一天产出多少图、多少视频？

**答**："工作流本身 7×24 不停跑，瓶颈在 API 配额和成本。按 5dock 当前定价，4 张候选图大概 ¥1-3，一条 15 秒视频大概 ¥10-30。如果你们 API 余额管够，理论上一天可以跑几百个 SKU。但**业务上**，4 选 1 之后审核员还要把关，审核员一天能审 50-100 个 SKU 是合理上限。所以瓶颈在审核，不在生成。"

### Q6：保修期之后怎么办？要再续一年得多少钱？

**答**："保修期满后有两种合作模式：a) 按工单计费，5000 / 工作日，做啥算啥；b) 包月运维（推荐），每月固定费用覆盖一定工时上限的小幅调优、配置维护、答疑。具体价格我们在保修期结束前 2 周给报价。如果你们届时不再需要支持，系统继续自己跑没问题——不存在'乙方不维护就死机'的设计。"

### Q7：价格能不能再低一点？

**答**："这个项目的成本主要是 25 - 30 个工作日的工程时间。我们给的报价是行业的中位偏低水平，因为这是一个重模板、轻定制的项目，乙方有规模效应。如果预算紧张，可以考虑：a) 暂时只做图片链路，砍掉视频，工期减到 12-15 天；b) 砍掉 ASR 字幕环节用现成 SaaS（节省 2-3 工作日）。我建议保留全链路，因为视频是后面 ROI 最高的部分。"

### Q8：交付物的归属权和源代码所有权？

**答**："交付的所有 N8N 工作流 JSON、Prompt 模板、文档、初始化 SQL、部署脚本，知识产权归你们公司。我们交付时会把整个 git 仓库 push 到你们指定的 Git 服务（GitHub / Gitee / 自建 GitLab 都行），从那一刻起你们自由处置。乙方保留'内部经验沉淀'的权利（不带客户数据、不带客户 prompt 模板的通用方法论），不会再用相同代码服务你们直接竞争对手。"

### Q9：迭代节奏？以后有新需求多久能上？

**答**："保修期内的小幅调优 1-3 个工作日响应。保修期外的小需求（< 1 工作日）按工单 24-48 小时响应；中等需求（3-5 工作日）按周响应；大需求（> 5 工作日）按月排期。我们建议你们指定一个内部产品对接人，避免多头沟通。"

### Q10：AI 生成的逻辑可解释吗？万一审核员问'为什么这张图选了这个模特'？

**答**："每张候选图都会归档它当时的完整 prompt（在 archive 表 prompt_text 字段，也会带在交付包里）。审核员可以倒查'这张图是因为 prompt 里写了 XXX 才长这样的'。模型本身的内部决策不可解释——这是 LLM / 扩散模型的本质——但 prompt 是可追溯的。这对版权、合规、内审的诉求基本够用。"

### Q11：安全性怎么样？我的 API key 会泄漏吗？

**答**："所有密钥只存在你们 VPS 的 `.env` 文件里，权限 600，只有 root 和 docker 进程能读。N8N 和 Postgres 都不暴露公网，只走 Caddy 反代 + 域名鉴权。OSS 不开公开读，访问全走 Access Key 签名 URL。我们交付后不持有任何凭据，离开后我们就没法访问。如果担心保修期内乙方接触敏感数据，可以保修期内为我们开一个临时跳板账号，用完即删。"

### Q12：如果做完不满意可以退款吗？

**答**："合同分阶段付款：开工 30% / 阶段 B 完成 30% / 阶段 D 验收 30% / 保修期满 10%。每个节点都有明确验收清单（参见 SOW §4），任一节点不通过就不进入下一节点付款。最坏情况是阶段 B 验收失败，那时你们已付 30%，乙方已交付一个能跑空架子，双方按工时结算后终止。完整退款条款写进合同附件 D，今天可以单独过一遍。"

## 3. 演示翻车应急预案

### 3.1 API 超时 / 模型返回失败

**症状**：飞书录入后 60 秒过去，候选图没出来。

**应对台词**："模型 API 偶尔会因为高并发排队，这种情况在生产中我们用重试机制处理（指 N8N 重试节点）。但今天演示我们就用兜底——这是上周已经为这次演示预生成好的同款 SKU 的 4 张候选（切到 OSS `_demo_seed/` 目录展示）。生产环境一旦出现这种超时，会自动重试 3 次，3 次都失败才告警，告警频率不到 1%。"

**操作**：切到 OSS `YN-BRA-001/_demo_seed/`，把 4 张图的链接手动写入飞书一行作为"刚刚生成"。

### 3.2 生成质量差到不能看

**症状**：4 张图都长得很怪（模特畸形、产品错乱）。

**应对台词**："这就是真实生产中的 4 选 1 场景——AI 不保证每张都好，所以我们设计了 4 倍冗余 + 审核 + 二次生成。我现在演示二次生成（操作）：审核员选驳回类别 + 写改向，新一轮就来了……（如果新一轮还差）这就是为什么样品验收阶段我们留了 5 工作日的调优时间，prompt 模板会在那期间收口。今天的演示侧重流程，不是最终调优后的效果。"

### 3.3 网络卡 / 飞书加载不出来

**症状**：飞书多维表 spinner 转半天。

**应对台词**："网络抖动，我切到本地预录的演示视频。"

**操作**：开桌面 `demo-fallback.mp4`，是上周完整跑通的录屏，9 分钟，可全程当备播。

### 3.4 N8N 节点崩溃 / 报错红框

**症状**：某节点变红、画布上有 ERROR。

**应对台词**："这是 N8N 给我们看的执行日志（点开节点）—— 你看，错误信息直接定位到第 X 行 prompt 里有个未转义的引号。生产中我们对所有 webhook 入口都做了 schema 校验，这种就根本进不来。今天为了演示快，我跳过了校验。"

**操作**：复制 prompt，去除特殊字符，重跑。如果还是不行，切兜底素材。

### 3.5 客户问到一个我不会答的问题

**应对台词**："这个问题挺关键，我现在不想给你一个不严谨的回答。我会议后 24 小时内书面答复，附上对照参考。"

**别做的事**：不要硬猜、不要搪塞、不要承诺范围外的功能。
