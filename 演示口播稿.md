今天演示二十分钟，肯定跑不完整条链路，所以我会同时做两件事：一边在你们眼前真实地启动一个新的任务，让大家看流程是真的在动；一边给你们看我昨天预先跑出来的成片，节省时间。

## 第 2 段 · 整体架构（3 分钟）

**【时长 3 分钟。切到 architecture.md 渲染页，把架构图那块滚出来。鼠标可以在图上点几下当指示。如果架构图加载不出来，直接口述也行。】**

先花三分钟讲一下整套系统长什么样，这样后面看流程的时候你们就知道每一步在哪里发生。

整套系统就五个角色，没有更多。

**第一个，飞书。**这是入口，也是审核台。你们运营在飞书多维表里录 SKU，审核员在另一张飞书表里点通过、点打回。整个交互对人的部分都在飞书里完成，不用学新工具。这一点我们是故意的——上线第一天运营就能上手。

**第二个，N8N。**这是工作流大脑，跑在你们 VPS 上一个 docker 容器里。你们可以理解成一个可视化的流程编辑器，每一步是一个方块，方块之间连线。后面我会让大家看一眼真实的画布。

**第三个，模型 API。**两路：图片走火山方舟的 Seedream 4.0，视频走火山方舟的 Seedance 2.0，文案和提示词改写走的是 Claude Sonnet 4.6，通过 5dock NewAPI 网关转发。模型这一层是解耦的——以后想换一个更便宜的图片模型，改一个端点就行，业务流程不动。

**第四个，存储。**素材本身放阿里云 OSS，元数据放 Postgres。所有路径都按 SKU 分目录，找东西很直接。

**第五个，反向代理 Caddy。**对外只暴露 80 和 443 两个端口，自动 HTTPS。N8N 的管理后台、Postgres、Redis 全跑在 docker 内网，外面访问不到。

**这里有一个重点要强调一下——**所有数据都在你们自己的 VPS、你们自己的 OSS、你们自己的飞书里。我们交付完之后，密钥也只在你们 .env 文件里。**我们不持有任何凭据，不持有任何素材副本，所有权 100% 是你们的。**

讲完了，五个角色：飞书、N8N、模型、存储、反代。下面我们看真实运转。

## 第 4 段 · 看图（4 分钟）

**【时长 4 分钟。屏幕停在飞书审核工作台，YN-BRA-001 这一条展开，显示十二张候选图。】**

这就是上一轮跑出来的候选图。一共十二张——四个场景，每个场景两张商品特写加一张人物场景图。

我先讲一下排版。每张图右边一列是 AI 自动打的分数，四个维度：品牌一致性、卖点呈现、真实感、合规性，各 0 到 10 分。**这张瑜伽馆场景的，您看，9.2，这是 Top-1。**面料的网眼质感、拉链的金属反光、模特的动作、构图——我们对照过你们提供的样品图，质感方向是对的。

再往右一列是审核操作——通过、打回、二次生成。我点一下"通过"试一下。**【点通过。】**飞书这里马上变绿，OSS 后台那边这张图就被从 raw 目录复制到 approved 目录里去了，Postgres 那边状态也跟着变。这是同步的。

我再演示一下打回。**【点另一张稍差的。】**这张拉链头颜色我看着不太对，本来应该是哑光黑，跑出来有点偏银。我选打回，类别选"产品细节不符"，备注栏写"拉链头改哑光黑，其他保留"。

**这里要重点讲一下——我们的打回不是一句话扔回去就完事。**审核员写的备注是结构化的——告诉系统要改什么、要保留什么。N8N 监听到这个事件，会把这两段反馈喂给 Claude 改写 prompt，再去 Seedream 重跑。这就避免了那种"AI 改完反而离要求更远"的常见问题。

打回有硬性上限——同一张图自动二次生成最多两轮，超出来就锁定，等人工介入。**这是为了防无限烧钱，每个 run 在数据库里都有 token 累计，超过预设阈值会自动告警。**

最后讲一下视频。

**【切到 OSS 控制台或者预先打开的播放器，播放 12 秒成片。】**

这条 12 秒视频是四个分镜拼出来的：开场两秒产品出现、三秒拉链特写、四秒跑步动态、三秒 logo 收尾。声音、节奏、字幕都已经在分镜脚本里规划好了。

**有一个细节请大家特别注意——**视频里的模特，和刚才图片里那张 9.2 分的，是不是一眼能看出是同一个人、同一个妆面、同一个发型？这就是我们做的跨链路一致性。原理是图片和视频共用同一份 SKU 解析结果、同一份品牌风格 token、同一组模特参考图。换其他品牌、换其他 SKU，这个机制都成立。

**——翻车口径——如果今天图片或者视频出来的效果就是看不下去：**

这恰好是真实生产里"四选一"机制存在的原因——AI 不保证每张都好，所以我们设计了四倍冗余加审核加二次生成。我现在演示二次生成给你们看。**【正常走打回流程。】**样品验收阶段我们留了五个工作日专门做调优，prompt 模板会在那段时间收口。今天的演示侧重看流程，不是看终态调优后的效果。

## 客户问 + 标准答复（八问）

> 不要照着背，只是把语气和论点准备到位。客户怎么问、就用对应那一条接住。

### Q1：AI 会不会幻觉？比如把拉链画反了、款式画错了？

A：这个我们做了三层防御。**第一层**，每张候选图生成的时候都强制以你们提供的白底产品图作为 reference，模型是"看着实物画"的，不是凭空编。**第二层**，prompt 里硬编码产品的关键特征——前拉链、黑色、高弹——这些字段不允许模型擅自换。**第三层**，审核员把关，"拉链头颜色不对"这种细节驳回会进入结构化的二次生成，不是自由文本。三层叠加之后，我们样品调试期实测幻觉率能压到 10% 以下，剩下的由四选一加审核兜底。我承诺的是"贴近实物、审核高效拦截偏差"，不承诺"零幻觉"——零幻觉目前是行业的物理上限，没人能做到。

### Q2：版权问题，AI 出的图我能商用吗？

A：可以。我们用的 5dock 转发的是商用授权模型，火山方舟 Seedance 也是商用授权，生成内容版权归调用方——也就是你们公司。交付的时候我会附一份《模型使用条款摘要》，每一个模型可商用范围都写清楚。**但是有一种情况要单独走法务**——如果你们要拿生成图去申请外观专利或者商标，那就不在常规商用范围里了。日常电商详情页、营销素材、社交媒体投放，没问题。

### Q3：能不能换更便宜的模型？我看市面上图片生成挺多家的。

A：可以换。架构是模型解耦的——只需要在 N8N 的图片提交节点改一个端点和参数映射，业务流程不动。视频同理可以换 Kling、可以换 Runway。**但是首期打样我建议先用 Seedream 4.0 把效果天花板打出来**，验收通过之后再讨论降本替换。先确认能做到、再讨论便宜——这个顺序比较稳。如果开局就用低端模型，万一效果不达标，原因到底是模型不行还是 prompt 不行就分不清了。

### Q4：万一火山或者 5dock 挂了怎么办？整条线就断了？

A：每个外部调用都设了三档保护——**超时 60 秒、自动重试 2 次、最终失败写入飞书告警表通知运营**。整个 run 不会因为某一步失败而丢，会停在失败的那一步等恢复。Postgres 里 run 状态机是完整的——任何时候断点续跑都可以。如果是火山长时间挂——这种情况一年也不会有几次——你们可以选择等火山恢复，也可以临时切到我们前面讲的备用模型。架构上这一切换大概十分钟工时。

### Q5：审核打回会不会无限烧钱？我担心实习生天天点打回。

A：硬性上限——**同一个候选物自动二次生成最多 2 轮，超出锁定状态等人工介入**。每个 run 在 Postgres 里有 token 和调用次数累计，超过预设阈值——比方说单 run 超过 30 块——自动告警。**做不到意外账单这种事**。另外审核员的打回操作本身不消耗模型调用，只是一个状态标记——只有"二次生成"这个动作才真的去重新跑模型。所以实习生狂点打回不会烧钱，狂点二次生成才会，那个有 2 轮上限。

### Q6：换品类——我们后面想做瑜伽裤、运动 T、运动外套——是不是要重做？

A：不需要重做。**流程零改动、配置在飞书填、只多写一份品类提示词，半天搞定。**第二款产品的接入预计 0.5 到 1 个工作日，第三款及以后通常 2 到 4 小时。原因是我们这次交付的不是一次性的内容产出，是一套抽象好的模板能力。**这是这个项目最大的价值，请务必记住这一点**——边际成本极低。

### Q7：数据安全和私有化？我的产品图、我的运营数据会不会跑到你们那儿？

A：所有数据落你们自己的 VPS、自己的阿里云 OSS、自己的飞书——**我们这边不持有任何素材副本，也不持有任何凭据**。N8N 是自托管的，工作流定义你们随时可以导出。密钥只在你们 .env 文件里，权限 600，只有 root 和 docker 进程能读。我们交付完之后就没法再访问你们的系统了。如果保修期内你们担心，可以为我们开一个临时跳板账号、用完即删——这个我们之前在别的客户那边做过。

### Q8：交付完之后我们想自己改怎么办？保修期之外谁支持？

A：N8N 是低代码——你们运营懂一点逻辑就能改节点参数。代码层面我们交付 docker-compose、prompt 模板、Postgres 初始化 SQL、部署脚本、运维手册，所有东西都 push 到你们指定的 git 仓库——GitHub、Gitee、自建 GitLab 都行。从交付那一刻起你们自由处置。**保修期一个月内**，bug 免费修，参数级调优也免费，每月累计三个工作日以内的小调整都包含。**保修期之后**两种方式：按工单计费——5000 一个工作日，做啥算啥；或者包月运维。具体价格保修期结束前两周给报价。**不存在"乙方不维护就死机"的设计**，这个请放心。

## 演示翻车通用兜底口径

> 不写在念稿里。Joshua 心里有数就行。
